# software-factory-template

A reusable starting point for shipping a new software project fast, built from what we actually learned completing Project #001 (missed-call-revenue-calculator). The goal is to cut the time to ship Project #002 by at least half by reusing the process, not just the code.

## What this is

This is not a code framework. It is a documentation and process template: the files and checklists a project needs from day one so that research, decisions, verification, and distribution never get skipped or reinvented from scratch.

## How to use it

1. Create a new repository from this template (or copy these files into a new repo).
2. Rename and fill in `templates/CHANGELOG.template.md` and `templates/DECISIONS.template.md` for the new project.
3. Work through `checklists/competitor-research-checklist.md` before writing any code.
4. Build the product.
5. Work through `checklists/verification-checklist.md` before calling anything done.
6. Work through `checklists/deployment-checklist.md` to ship it.
7. Work through `checklists/release-checklist.md` before any public/distribution step.
8. Use `templates/REALITY_REPORT_TEMPLATE.md` every time a real user touches the product.
9. Use `checklists/post-release-experiment-checklist.md` to decide what to measure and what to try next.

## Structure

- `templates/` — fill-in-the-blank documents every project should have.
- `checklists/` — step-by-step gates to pass through at each stage of the factory.

## Guiding principle

Project #001 proved the pipeline works: research, build, verify, ship. The objective now is not to keep improving Project #001 indefinitely — it's to make every future project faster and more reliable by improving this factory instead.
